01. Що таке MLOps та AIOps

Натренувати модель — це 10% роботи. 90% — це підтримати її роботу в продакшні: дані оновлюються, моделі дрейфують, інфраструктура падає, бізнес змінює правила. MLOps і AIOps — про те, щоб ця підтримка була systematic, а не "пожежна".

🛠 MLOps

Machine Learning Operations. Practices + tools для життєвого циклу ML моделей: data → train → deploy → monitor → retrain. Аналог DevOps, але з додатковими рівнями: version data, version models, monitor drift, manage experiments.

🤖 AIOps

AI for IT Operations. Використання ML/AI для моніторингу і керування інфраструктурою. Anomaly detection у логах, root cause analysis, incident prediction, auto-remediation.

Чому "просто DevOps" не вистачає

critModels silently degrade

Код не змінюється — а accuracy впала з 91% до 73% за місяць. Дані dрейфують, ти не бачиш цього в логах сервера.

critReproducibility

"Ми тренували це 3 місяці тому" — а train data вже інший, library version інша, hyperparameters в чиємусь notebook.

medExperimentation велика

Data scientists запускають 200 експериментів. Хто переміг? Як порівняти? Куди підключити winner?

medData versioning

Train на v1 → deploy → v2 даних → нова модель не bit-equal. Без data version — нема reproducibility.

medTrain/serve skew

Pipeline featurization training відрізняється від production. 5% accuracy gap "де? чому?" — і це норма без feature store.

lowLong feedback loops

Помилкова credit decision виявиться через 6 місяців через chargeback. До того часу модель уже випустила 100K bad predictions.

02. DevOps vs MLOps vs LLMOps

Це не просто три різні слова — це три різні дисципліни з різними артефактами, тестами і failure modes.

🛠 DevOps
  • Артефакти: код, контейнери
  • Тести: unit, integration, e2e
  • Версіонування: Git
  • Deploy: blue/green, canary
  • Моніторинг: latency, errors, CPU/RAM
  • Failure mode: exception, 500
  • Cycle: commit → CI → CD → prod
📊 MLOps
  • Артефакти: код + дані + модель
  • Тести: data validation, model perf, fairness
  • Версіонування: Git + DVC + MLflow registry
  • Deploy: shadow, A/B, champion-challenger
  • Моніторинг: + accuracy, drift, bias
  • Failure mode: silent degradation
  • Cycle: data → train → eval → deploy → monitor → retrain
🤖 LLMOps
  • Артефакти: prompts + RAG + agents + few-shot
  • Тести: eval sets, LLM-judge, regression
  • Версіонування: prompt registry (Langfuse, PromptLayer)
  • Deploy: feature flags, prompt A/B
  • Моніторинг: hallucinations, токени, $$$
  • Failure mode: впевнено невірна відповідь
  • Cycle: prompt → eval → deploy → trace → iterate

Що робить ML "складнішим за код"

ДисциплінаЩо версіонуєтьсяЩо тестується

LLMOpsPrompts + tool definitions + retrieval + model versionQuality of generation на eval set + production traces

03. ML Lifecycle — повний цикл

Це не "стрілочка зліва направо" — це безкінечний цикл. Як тільки модель потрапила в прод, починається monitor → drift → retrain → redeploy. Більшість команд недооцінюють цикл і релізять "одноразово".

04. MLOps Stack — інструменти за рівнями

Стандартного стеку нема — є category інструментів. Маленька команда обере 3-4, велика — 10+. Це шар, який треба знати щоб обирати свідомо.

📥 Data layer
Збір, валідація, версіонування даних. Tools: DVC, LakeFS, Great Expectations, Pachyderm. Storage: S3, GCS, Snowflake, BigQuery. Streaming: Kafka, Kinesis.
🔧 Feature Store
Спільні фічі для train і serving. Tools: Feast, Tecton, Hopsworks, Vertex AI Feature Store. Розв'язує train/serve skew.
🧪 Experiment Tracking
Версії, метрики, артефакти експериментів. Tools: MLflow, Weights & Biases, Neptune, Comet. ClearML для full-stack.
📦 Model Registry
Зберігання версій моделей зі стейджами (staging/prod/archived). MLflow Registry, SageMaker Model Registry, Vertex AI Model Registry, BentoML.
🚀 Serving / Deployment
Експонування моделей як API. Tools: BentoML, Seldon, KServe, TorchServe, Triton. Cloud: SageMaker Endpoints, Vertex AI, Databricks Model Serving.
⚙ Orchestration / Pipelines
Workflow для train, batch inference, retraining. Tools: Airflow, Prefect, Dagster, Kubeflow Pipelines, Metaflow, Flyte.
📊 Monitoring & Observability
Drift, perf, data quality в проді. Tools: Evidently, Arize, WhyLabs, Fiddler, NannyML. Для LLM — Langfuse, LangSmith, Helicone.
🏷 Labeling & Annotation
Розмітка нових даних з продакшну для retraining. Tools: Label Studio, Snorkel, Scale AI, Argilla (LLM-focused).
📜 Governance & Lineage
Audit, compliance, model cards, fairness reports. Tools: OpenLineage, DataHub, Amundsen. Critical для regulated industries (banking, healthcare).

Як обрати — мінімальний практичний MLOps стек

🚀 Startup MVP (2 інженери)

Git + DVC + MLflow + FastAPI + Prometheus. Без feature store, без Kubeflow. Працює до 5-10 моделей у проді.

🏢 Mid-stage (5-10 ML)

+ Airflow / Prefect, + Evidently, + Feast. Auto-retraining piepeline, drift alerts. ~25-50 моделей.

🏛 Enterprise (50+ ML)

SageMaker / Vertex AI full suite, Tecton, Arize, OpenLineage. Compliance audits. Хтось окремий тримає платформу.

05. CI/CD для ML — від коміта до прод-моделі

У звичайному CI/CD: код → тести → deploy. У ML: код + дані + модель → data validation + model tests + fairness checks + shadow mode → deploy. На кожен етап є gate, який може заблокувати release.

Strategies деплою моделі

👁 Shadow mode

Нова модель отримує prod traffic, але виходи не повертаються користувачам — лише логуються для порівняння. Безпечно вимірюємо delta.

🚀 Canary

5% → 25% → 50% → 100%. На кожному кроці monitoring. Якщо метрика просіла — auto-rollback.

⚔ Champion-Challenger

Champion (current) обслуговує 80%, challenger (new) — 20%. Через тиждень дивимось хто переміг по бізнес-метриках.

🔵 Blue-Green

Дві ідентичні інфраструктури. Деплой на blue, тестуємо, перемикаємо traffic. Швидкий rollback.

🅰 A/B test

50/50 для нової фічі. Не для безпеки — для гіпотези про бізнес.

🎚 Multi-armed bandit

Динамічна ваги між моделями. Dynamic exploration vs exploitation. Складно але оптимально.

06. Monitoring & Drift Detection

Ось де "ML отличається від коду". Сервіс зеленый — pings ОК, latency нормальна, errors=0. А accuracy впала на 18% за місяць, бо змінився user behavior. Без spec drift detection ти просто не побачиш цього.

Типи drift

critData drift

Distribution input features змінилася. P(X) у проді ≠ P(X) на трейні. Наприклад: новий регіон, інша демографія, новий продукт.

critConcept drift

Зв'язок X→Y змінився. P(Y|X) інше. Наприклад: covid → люди купують маски а не парфуми. Найскладніший вид.

medLabel drift

Distribution Y змінилась. Раніше fraud rate 1%, тепер 5%. Class imbalance ламає precision/recall.

medPrediction drift

Output моделі шифтиться. Може бути сигналом data drift до того як упала accuracy. Дешева перевірка без labels.

lowSchema drift

Поле зникло, тип змінився, новий enum. Ламає pipeline. Великі data validation tools (Great Expectations) ловлять.

lowPerformance drift

Latency p95 ↑ через більший batch size або складніший feature set. Не data, але пов'язано.

07. AIOps — AI для управління інфраструктурою

AIOps — це не "ML моделі в проді" (це MLOps). Це застосування ML/AI до проблем DevOps/SRE: anomaly detection у метриках, root cause analysis по логах, prediction інцидентів, auto-remediation.

4 основні категорії AIOps

🔍 Anomaly Detection

ML моделі (Isolation Forest, LSTM Autoencoders, Prophet) знаходять незвичайну поведінку у time series метрик. Краще за статичні thresholds — адаптивні до seasonality.

🧠 Root Cause Analysis

Коли pager paged — графа сервісів + correlation analysis + log clustering показує "ось 3 ймовірні причини". Скорочує MTTR з 60 до 12 хвилин.

📈 Predictive Alerting

Замість "disk full at 95%" → "disk filling at this rate, will be full in 2h". Час реагувати proactively, а не fire-fighting.

🤖 Auto-remediation

Найпопулярніші incident patterns → автоматичні runbooks: scale up, restart, rollback, failover. Людина залучається тільки якщо bot не справився.

AIOps tools

Datadog Watchdog / AIOps

Auto-detect anomalies на metrics, logs, traces. Forecasting. Корелює incidents.

New Relic AI

Apdex anomaly, error grouping, golden signals. AI assistant для SRE.

Dynatrace Davis

Causal AI engine. Сам будує service map, automatic root cause.

BigPanda / Moogsoft

Alert correlation і noise reduction. Зменшує 1000 alerts → 10 incidents.

PagerDuty AIOps

Intelligent alert routing, на-каллер'у показує тільки критичні.

Open Source

Prometheus + Cortex anomaly, OpenSearch ML, Robusta для k8s.

08. Incident Response для ML/AI систем

ML інциденти — інші. "Сервіс не падає, але результати шкідливі". Hallucination, bias, data poisoning, prompt injection. SRE runbook потрібен інший.

ML Incident Runbook

1
Detect

Alert fired (drift, accuracy, error rate, customer complaint). severity level (P1-P4).

2
Triage

Подивитись dashboard. Чи інші моделі теж? Чи це конкретно ця? Чи це data issue?

3
Mitigate

Швидкий fix: rollback до previous version, або fallback rule (rule-based замість ML), або circuit-breaker (return default).

4
Investigate

Логи, traces, sample bad predictions. Що в input? Що в output? Корелює з deployment?

5
Fix root cause

Retrain з новими даними, fix bug у feature pipeline, update threshold. Не на гарячу в проді — через CI/CD.

6
Postmortem

Blameless. Що сталось, чому, що зробити щоб не повторилось. Action items з owner-ами і deadlines.

ML-specific failure modes

critSilent accuracy drop

Сервіс ОК, але передбачення гірші. Detect: drift monitors + rolling accuracy on labeled feedback.

critData poisoning

Хтось зіпсував train data (deliberately або bug). Симптом: model bias на specific groups.

critTrain/serve skew

Featurization різна. Прогнози ОК на test, плохі в проді. Feature store вирішує.

medCold start

Новий user/item немає історії — рекомендації безглузді. Fallback на popularity-based.

medAdversarial attacks

Spam-фільтр обходять обфускацією. Image classifier — adversarial patches. Robust training.

medBias amplification

Модель посилює existing biases у даних. Fairness monitoring + balanced sampling.

09. MLOps метрики — що міряти

Категорії метрик

💼 Business metrics

Conversion uplift, revenue per user, churn reduction, fraud savings. Це те, заради чого існує модель. Якщо не корелює з accuracy — accuracy неважлива.

📊 Model metrics

Accuracy, F1, AUC, RMSE для класичного ML. Hallucination rate, faithfulness, BLEU/ROUGE для LLM. Cohen's kappa для labeled feedback.

⚙ Infra metrics

P50/P95/P99 latency, throughput (RPS), GPU utilization, memory, $cost per 1K predictions. Cold start time для autoscaling.

📈 Drift metrics

PSI, KS p-value, feature distribution distance, prediction distribution shift. Per-feature і aggregate.

🔄 Pipeline metrics

Train pipeline success rate, deploy frequency, MTTR, rollback frequency. Це DORA-like для ML.

⚖ Fairness metrics

Demographic parity, equalized odds, predictive parity. Disparate impact ratio. Critical для regulated industries.

10. Pitfalls — що часто ламається

90% фейлів MLOps — не технічні, а організаційні. "ML team не говорить з infra", "data scientists не розуміють CI/CD", "немає budget на платформу до того як 5 моделей у проді ламаються".

crit"Деплоїмо коли модель готова"

А потім три місяці нічого. Model rotting. Бажано auto-retrain pipeline з самого старту.

critTrain/serve skew

Pandas у train, SQL у serving. 5% accuracy gap "де?". Feature store або shared feature library — обов'язково.

critНемає data validation

Pipeline ламається коли data team змінив schema. Great Expectations або equivalent — на кожен ingestion.

critNotebook → prod direct

"Він працює локально". Жоден тест не пройшов. Config — у комірках. Виправляється культурою + linters + pre-commit hooks.

medОдин моніторинг для всіх моделей

Generic dashboard показує усе, нічого не зрозуміло. Per-model SLOs, per-model alerts.

medБез data lineage

"Звідки ця колонка взялась?" — ніхто не знає. OpenLineage / DataHub дають історію змін на даних.

medRetraining без trigger logic

"Кожен понеділок переучуємо" — навіть коли drift = 0. Або retrain on drift, або hybrid (schedule + drift).

lowVendor lock-in

Усе на SageMaker → не можеш перейти. Layer abstraction (BentoML, MLflow) дає portable models.
